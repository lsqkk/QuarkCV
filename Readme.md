A polished, data-driven XeLaTeX resume template for academic CVs, research resumes, and portfolio-style one-pagers.

QuarkCV turns structured YAML into a clean, modern, link-rich PDF, while keeping your content source separate from your layout system.

QuarkCV is a YAML-powered XeLaTeX resume template with a polished academic demo, clickable PDF links, local-first avatar support, and a clean separation between content and presentation.

![QuarkCV Demo](Demo/resume_demo.png)

## At a Glance

- Public demo repository with fictional academic sample data
- `Data/*.yml` as the single source of truth
- `main.tex` generated automatically from structured content
- local-first avatar loading with remote fallback
- clickable links across contact info, institutions, projects, and publications
- suited for academic CVs, research resumes, and template forks

## Why QuarkCV Works Well

- Data first. Resume content lives in `Data/*.yml`, not scattered across TeX blocks.
- Resume-aware fields. Long-form archive content can coexist with concise resume-specific fields such as `resume_title`, `resume_summary`, and `resume_highlights`.
- Clean separation of concerns. `Data/*.yml` stores content, `scripts/build_resume_data.py` builds the document, and `myresume.sty` controls visual design.
- Clickable PDF output. Email, phone, homepage, school links, project links, publication links, and local files are generated as clickable links.
- Demo-ready out of the box. The repository already includes a complete academic sample profile, so the template is understandable immediately after cloning.
- Safe fallbacks. Avatar loading prefers local files, supports remote URLs as an optional fallback, and gracefully falls back to an initials-based placeholder.

## How It Works

1. Edit the data files in `Data/`.
2. Run `python scripts/build_resume_data.py`.
3. The script generates `main.tex` directly.
4. Compile `main.tex` with XeLaTeX.

In this workflow:

- `Data/*.yml` is the source of truth.
- `main.tex` is generated output.
- `myresume.sty` is the visual system.

## Quick Start

### 1. Edit your data

Update the YAML files in `Data/`:

- `Data/_config.yml`: identity, theme, contact links, section visibility, preview counts, avatar
- `Data/profile.yml`: summary, research focus, skills
- `Data/education.yml`: education entries and compact highlights
- `Data/project.yml`: projects, resume titles, resume summaries
- `Data/awards.yml`: awards and short reasons
- `Data/publications.yml`: publications and publication links

### 2. Generate `main.tex`

```bash
python scripts/build_resume_data.py
```

### 3. Compile the PDF

```bash
xelatex main.tex
```

Running XeLaTeX twice is recommended for a clean final output.

## Demo Data Philosophy

This template separates archival detail from resume-facing presentation.

For example, a project can contain:

- `title`: the full original title
- `description`: a long-form homepage-style description
- `resume_title`: the shorter title used in the resume
- `resume_summary`: the compact version optimized for resume reading

The same pattern is used across the template:

- `profile.yml` uses `resume_summary` and `resume_focus`
- `education.yml` uses `resume_summary` and `resume_highlights`
- `project.yml` uses `resume_title` and `resume_summary`

This lets one dataset support both richer personal-site content and concise resume output.

## Avatar Logic

Set `avatar` in `Data/_config.yml` to one of the following:

- a local relative path such as `./Data/avatar.png`
- a remote `http` or `https` image URL
- an empty value

Avatar resolution works like this:

1. Prefer a local file path if the file exists.
2. If the value is a remote URL, try downloading and caching it locally.
3. If neither works, show a clean initials-based placeholder avatar.

This keeps the template stable in offline or demo environments while still supporting remote assets.

## Recommended Workflow

- treat `Data/*.yml` as your content layer
- treat `scripts/build_resume_data.py` as your builder
- treat `main.tex` as generated output
- treat `myresume.sty` as your design layer

That separation keeps the project easy to extend without turning the TeX file into a maintenance burden.

## Notes

- Use XeLaTeX for compilation.
- `main.tex` is generated automatically, so manual edits will be overwritten.
- If you update avatar paths or local linked files, regenerate before compiling.
- The default profile in this repository is fictional and intended only as demo content.

## FAQ

### Can I use this for a real personal resume?

Yes. Replace the demo data in `Data/*.yml` with your own information, regenerate `main.tex`, and compile the PDF.

### Do I need to edit TeX directly?

Usually no. Most content updates should happen in `Data/*.yml`. The styling layer lives in `myresume.sty` if you want deeper visual customization.

### Can I use a local avatar only?

Yes. That is the preferred path for stable builds and public demo repositories.

### Can I extend this with new sections?

Yes. The current structure is intentionally simple and can be expanded with new YAML-backed sections such as internships, teaching, talks, grants, or service.
